## What Works in Ratatui That We Can Borrow

1. Product identity is immediately visible.
- Strong visual language and explicit technical positioning appear above the fold.

2. Demo-first experience is obvious.
- "Try live demo" and embeddable widget paths are surfaced early.

3. Proof posture is paired with style.
- Determinism/performance claims are visible alongside visual polish, not hidden in deep docs.

4. Multiple audience paths are present.
- Terminal users, web users, and React integrators each get a clear entry point.

## What Works in Charm-Style Developer UX

1. Aesthetic terminal output is treated as core product quality.
2. Tooling supports repeatable media capture (`vhs`) instead of ad-hoc screen recording.
3. Small composable pieces (styles/forms/components) keep visual upgrades incremental.

## Constraints Specific to Vifei

1. Truth-path invariants cannot be compromised.
- Showcase mode must not alter EventLog truth, ordering, or artifact semantics.

2. Demo mode must remain deterministic.
- Same fixture + same flags should still produce stable proof artifacts.

3. Showmanship belongs to projection and capture layers.
- Styling, layout emphasis, and scripted demo pacing are allowed in UI.
- Core reducer/projection truth mechanics remain unchanged.

## Recommended Showcase UX Profile (v0.1.1 Scope)

### Profile Name

`--profile showcase`

### Profile Behavior

1. Visual emphasis
- richer border/header styling
- clearer section hierarchy for Incident/Forensic lenses
- stronger highlight treatment for "Next action" and refusal diagnostics

2. Demo readability
- width defaults tuned for recording/capture
- stable spacing rules for headline metrics and Truth HUD
- optional compact title banner for screenshots

3. No semantic drift
- same data, same ordering, same truth fields
- only presentation changes

### Output/Capture Integration

Add deterministic showcase captures to existing asset pipeline:

- `incident-lens-showcase.txt`
- `forensic-lens-showcase.txt`
- `truth-hud-showcase.txt`

Generated by existing capture flow, with explicit profile flag.

## Product Hunt and Launch Implications

From Product Hunt official guidance:

1. Launch quality favors authentic engagement, not vote farming.
2. First-comment clarity and concrete product explanation matter.
3. Complete listing assets and maker responsiveness matter.

Practical implication for Vifei:

- lead with one runnable command and one verifiable output
- keep media deterministic and reproducible
- avoid exaggerated claims that cannot be tested

## Phased Execution Plan

### Phase A (implement first)

1. Add `showcase` profile flag to TUI view model.
2. Apply style/layout-only changes in Incident/Forensic render paths.
3. Extend capture binary to emit showcase assets.
4. Add tests that prove no truth-path semantic drift under showcase profile.

### Phase B

1. Add `docs/showcase/` page content and asset references.
2. Add one scripted demo flow optimized for 45-90 second clips.
3. Add README section linking to showcase page and verification path.

### Phase C

1. Optional web embed experiments (separate branch) for interactive showcase.
2. Compare adoption impact before deeper UI architecture changes.

## Risk Notes

1. Over-styling can reduce clarity under narrow terminals; keep width-aware tests.
2. Style-only changes can accidentally drift layout assumptions; add snapshot checks.
3. Separate showcase page must not diverge from verified command behavior.

## Decision

Proceed with `showcase` profile implementation first, using style/layout-only changes and deterministic capture expansion.
